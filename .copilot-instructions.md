# Tilda - Development Guide

## Quick Reference

See [README.md](README.md) for setup, running, and basic usage.

## Domain Model

**What:** Shared car booking for a closed group
**Pattern:** Day-based reservations (one day = one booking unit)
**State:** Existence = state (no status fields)

## Architecture Principles

1. **Config over code** - resource details in `config.edn`
2. **One instance = one resource** - keep scope tight
3. **Curl-friendly API** - JSON responses, simple endpoints
4. **Server-driven UI** - Datastar reactivity, minimal JS
5. **Existence = state** - document exists = active, deleted = resolved
6. **Pure deciders** - conflict resolution logic is testable without DB
7. **XTDB for audit** - bitemporal history preserves all changes

## Data Model

Both tables use "existence = state" pattern. XTDB history preserves audit trail.

```clojure
;; Request - expression of interest
;; Exists = pending, deleted = resolved
{:xt/id #uuid "..."
 :tenant-name "Flo"
 :start-date #inst "2026-03-01T00:00:00Z"
 :end-date #inst "2026-03-03T00:00:00Z"
 :priority 0
 :requested-at #inst "..."}

;; Booking - confirmed reservation
;; Exists = confirmed, deleted = cancelled
{:xt/id #uuid "..."
 :request-id #uuid "..."  ; link to winning request
 :tenant-name "Flo"
 :start-date #inst "2026-03-01T00:00:00Z"
 :end-date #inst "2026-03-03T00:00:00Z"}
```

## Domain Functions (tilda.booking)

```clojure
;; Requests (existence = pending)
(create-request! node {:tenant-name start-date end-date priority})  ; idempotent for overlaps
(cancel-request! node id)
(get-request node id)
(pending-requests node)
(find-conflicting-requests node start end)
(request-history node id)                 ; XTDB time-travel

;; Resolution (atomic: creates booking, deletes all requests)
(resolve-slot! node requests decider-fn)

;; Bookings (existence = confirmed)
(get-booking node id)
(all-bookings node)
(find-conflicts node start end)
(cancel-booking! node id)
(booking-history node id)

;; Deciders (pure functions - no DB needed)
(decide-first-come-first-serve requests)
(decide-by-priority requests)
(decide-random-lottery requests)
```

## API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| GET | / | Home page |
| GET | /calendar | Interactive calendar |
| GET | /calendar/events | SSE live updates |
| GET | /requests | List pending requests |
| POST | /requests | Create request (idempotent) |
| DELETE | /requests/:id | Cancel request |
| POST | /requests/resolve | Resolve competing requests |
| GET | /bookings | List bookings |
| GET | /bookings/:id | Get booking |
| DELETE | /bookings/:id | Cancel booking |
| GET | /bookings/:id/history | Audit trail |

## Calendar UX

- Drag to select dates → POST /requests
- Click/drag starting on YOUR pending dot → DELETE (cancels it)
- SSE pushes updates to all connected browsers
- Infinite scroll loads more months lazily

## Not Yet Implemented

- **Auth** - `auth.clj` has stub, ready for Oak integration
- **Admin UI** - Resolve conflicts via UI (currently API-only)
- **Request extension** - Drag to extend existing request
- **Overlap visualization** - Show conflicts in calendar
- **Deployment** - Dockerfile + env var support for lab orchestrator
